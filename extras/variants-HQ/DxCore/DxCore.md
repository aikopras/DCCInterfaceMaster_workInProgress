# DCC Encoding on DxCore

This DxCore implementation targets AVR processors supported by **DxCore**, in particular devices from the **AVR DA**, **DB**, and **EA** families. It is intended for use with the DxCore Arduino environment.

## Supported processors and hardware requirements

This implementation supports DxCore-based AVR devices that provide the required timer peripherals, in particular:

- AVR DA series  
- AVR DB series  
- AVR EA series  

The implementation depends on two hardware timers:

- **TCA0** for DCC waveform generation  
- **TCD0** for the RailCom cutout on the AUX pin  

Because of this, both timers must remain available exclusively for the DCC driver and may not be used elsewhere in the application.

The pin requirements are also hardware-driven:

- **DCC output (rail signal):** this must be one of the valid **TCA0 waveform output pins**
- On DxCore devices, this means **pin 0, 1, or 2 of a port**
- **AUX output (enable / RailCom cutout):** this must be either **PA4** or **PA5**

The choice of DCC pin determines which TCA0 compare unit is used:

- `Px0` → compare unit 0  
- `Px1` → compare unit 1  
- `Px2` → compare unit 2  

In addition, DxCore requires that the TCA0 waveform outputs for the selected compare unit are routed through the correct port multiplexer setting. Only ports supported by the TCA0 port multiplexer can be used.

### Important notes

- **TCA0** is used as the DCC waveform generator and must not be used elsewhere in the program.
- **TCD0** is used for the RailCom cutout on the AUX pin and must not be used elsewhere in the program.
- The **DCC pin is not arbitrary**: it must be a TCA0 waveform output pin, which in practice means pin 0, 1, or 2 of a valid port.
- The **AUX pin is restricted to PA4 or PA5**, because those pins can be driven by the required TCD0 waveform outputs.

## Implementation details

### Timer-based DCC signal generation

The DCC waveform is generated by **TCA0** in **single-slope PWM mode**.

For each DCC bit, the driver programs:

- `PER` → the total bit time
- `CMPn` → the midpoint at which the output toggles

The timer overflow ISR is called after each complete DCC bit. In that ISR, the `PER` and compare register for the **next** bit are written. This means the timer hardware generates the actual waveform, while the ISR only prepares the next symbol.

DCC timing used in this implementation:

- Logical "1": **116 µs** total bit time
- Logical "0": **200 µs** total bit time

These are represented as:

- `PER = ONE`, `CMP = ONE / 2`
- `PER = ZERO`, `CMP = ZERO / 2`

The end bit is encoded as a logical "1".

Because the waveform is generated directly by TCA0 hardware, the DCC output itself is deterministic as long as the ISR updates the next symbol in time.

---

### Direct register updates

The TCA peripheral offers both buffered and unbuffered ways of updating `PER` and `CMP`. This implementation uses **direct, unbuffered register updates**.

The reason is synchronization. Buffered updates only become active after the current period completes, which would shift the timing relationship between the DCC packet boundary and the AUX RailCom cutout. Direct updates allow the next bit timing to take effect immediately for the next cycle and therefore make precise alignment possible.

The trade-off is that the ISR has less timing margin to prepare the next bit than in a buffered design. For this driver, that trade-off was accepted in order to keep the DCC and RailCom timing aligned.

---

### Packet-level ping-pong buffering

This implementation also uses a ping-pong mechanism, but not at the hardware waveform level.

Instead of copying timer symbols inside the ISR, two complete **packet buffers** are maintained:

- one buffer owned by `send()`
- one buffer owned by the ISR

The `send()` function prepares a complete packet in the currently free user buffer. Once complete, it marks that packet as ready. At the end of a transmission, the ISR switches to that newly prepared packet if one is available.

This design avoids `memcpy()` operations inside the ISR and keeps ISR execution time stable. The ISR only switches packet pointers and loads the next waveform symbol into the timer registers.

---

### Packet encoding

Packets are converted into arrays of timer symbols before transmission.

Each symbol consists of:

- `per` → total bit period
- `cmp` → compare point inside that period

Each packet may contain:

- an optional RailCom cutout symbol at the beginning
- preamble bits
- start bits
- data bits
- end bit

For RailCom on the DCC output, a dedicated gap symbol is inserted with:

- `PER = TCE`
- `CMP = TCS`

As in the other drivers, the preamble is shortened by four "1" bits when a RailCom cutout is inserted on the DCC stream.

---

### RailCom cutout on the AUX pin

The RailCom cutout on the AUX pin is generated by **TCD0**.

TCD0 is configured in **one-ramp mode**. The compare registers define:

- the moment when the cutout starts
- the moment when it ends

The AUX pin must therefore be either:

- **PA4** → TCD0 compare unit A
- **PA5** → TCD0 compare unit B

At the start of a new packet, the TCA-side logic can enable TCD0 so that the RailCom cutout is generated in parallel with the DCC packet timing. Once the cutout duration has completed, the TCD0 overflow ISR disables the timer again.

This part of the implementation is intentionally hardware-based as well, so the cutout timing does not depend on repeated software pin toggling.

---

### TCA0 configuration details

TCA0 is configured as follows:

- **single-slope PWM mode**
- **clock source = CLK_PER**
- **prescaler = 1**
- **overflow interrupt enabled**
- one compare unit enabled, depending on the selected DCC pin

The compare unit is selected from the chosen DCC output pin:

- port pin 0 → compare unit 0
- port pin 1 → compare unit 1
- port pin 2 → compare unit 2

The port multiplexer is then configured so that TCA0 outputs are routed to the correct port. After that, the timer registers are initialized and the selected DCC output pin is configured as an output.

---

### TCD0 configuration details

TCD0 is configured specifically for the AUX RailCom cutout:

- **clock source = CLK_PER**
- **synchronization prescaler = 4**
- **counter prescaler = 1**
- **one-ramp waveform mode**
- **overflow interrupt enabled**

The key timing values are:

- `TCD_TCS` → cutout start
- `TCD_TCE` → cutout end

These values are derived from the CPU clock and include a small compensation offset for ISR start latency.

The TCD waveform output itself is only attached to the AUX pin when RailCom-on-AUX is enabled. This allows the same AUX pin to be used both as H-bridge enable output and as hardware-generated RailCom cutout signal.

---

### Pin inversion and output polarity

The DCC signal polarity can be inverted by setting the inversion bit in the selected `PORTx.PINnCTRL` register for the DCC output pin.

The AUX output polarity is controlled separately. For AUX, inversion is applied through the pin control register of PA4 or PA5. This makes it possible to adapt the enable polarity of the external H-bridge without changing the rest of the timing logic.

---

## Signal quality

The **DCC signal** is generated directly by **TCA0 hardware**. The ISR only updates the timing parameters for the next bit. As a result, the DCC waveform itself should not show jitter, provided the ISR completes before the next transition must be prepared.

The **RailCom cutout on the AUX pin** is generated by **TCD0 hardware** and is therefore also hardware-timed. A small compensation offset is included for timer start latency. In practice, this means the AUX RailCom cutout should also be very stable, although its exact alignment depends on the start timing of TCD0 relative to the DCC packet boundary.

Overall, this design keeps both the DCC output and the AUX RailCom cutout primarily in hardware, with software only selecting packets and loading the next timing values.
