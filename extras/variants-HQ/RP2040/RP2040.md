# DCC Encoding on RP2040 / RP2350

This implementation targets the **RP2040** and **RP2350** families and is intended for use with the **Philhower Arduino core**, which was also used for testing.

## Supported processors and hardware requirements

This implementation supports:

- **RP2040**
- **RP2350**

The design is based on **PIO** (Programmable I/O) and **DMA**:

- one **PIO block**
- **two PIO state machines**
  - one for the DCC signal
  - one for the AUX / RailCom cutout signal
- one **DMA channel** to feed the DCC PIO state machine

The pin requirements are flexible:

- **DCC output (rail signal):** any GPIO pin that can be controlled by a PIO state machine
- **AUX output (enable / RailCom cutout):** any GPIO pin

This is one of the advantages of the RP2040 / RP2350 architecture: unlike timer-based implementations on some other processors, the pin choice is not tied to one specific timer output pin.

### Important notes

- This implementation was tested using the **Philhower board definition / Arduino core**.
- The DCC signal uses **one PIO state machine**.
- The AUX / RailCom cutout signal uses **a second PIO state machine**.
- The DCC waveform data is transferred to the PIO by **DMA**, so the CPU is not involved in bit timing once transmission has started.
- The AUX pin is optional. It can be used as a normal H-bridge enable pin, and optionally also as a RailCom cutout output.

## Implementation details

### PIO-based DCC signal generation

The DCC waveform is generated by a dedicated **PIO state machine**. The PIO program reads precomputed waveform symbols from its TX FIFO and outputs the requested pin levels with microsecond timing.

Each DCC symbol consists of two phases:

- level A + duration A
- level B + duration B

The symbol format is packed into a 32-bit value:

- bit 0: level A
- bits 1..15: duration A
- bit 16: level B
- bits 17..31: duration B

This is conceptually similar to the symbol-based approach used in the ESP32 RMT implementation, but here the symbols are executed by the RP2040 / RP2350 PIO hardware.

DCC timing used in this implementation:

- Logical "1": two half-bits of **58 µs**
- Logical "0": two half-bits of **100 µs**

Because the PIO executes the waveform autonomously, there is no interrupt-driven bit generation.

---

### Symbol-based packet encoding

Packets are translated into arrays of precomputed symbols before transmission.

Each packet may contain:

- an optional **start-of-packet marker**
- an optional RailCom cutout symbol in the DCC waveform
- preamble bits
- start bits
- data bits
- end bit

The implementation uses several predefined symbol types:

- logical "1"
- logical "0"
- end bit
- RailCom gap
- packet-start marker

The packet-start marker is a special symbol with value `0x00000000`. It is not part of the DCC waveform itself; instead, it is interpreted by the PIO program as a synchronization marker for the AUX RailCom state machine.

If RailCom on AUX is active, this marker is inserted before each packet. The marker slightly affects the end-of-packet timing, and the code compensates for that by adjusting the duration of the second half of the DCC end bit.

---

### DMA-driven waveform streaming

The PIO TX FIFO for the DCC state machine is fed by a **DMA channel**.

The DMA configuration is:

- transfer size: **32-bit**
- increment source address after each read
- do not increment destination address
- destination: **PIO TX FIFO**
- pacing: **PIO DREQ**

When a packet has been transferred to the FIFO, the DMA raises an interrupt. The DMA ISR does not generate waveform data. It only:

1. acknowledges the DMA interrupt
2. selects the next packet
3. programs the DMA with the next packet address and length

This keeps ISR execution short and deterministic.

---

### Packet scheduling and ping-pong buffering

This implementation uses packet-level ping-pong buffering.

Two packet buffers are maintained:

- one buffer owned by `send()`
- one buffer owned by the DMA ISR

The workflow is:

1. `send()` builds a complete packet in the inactive buffer
2. `send()` publishes that packet and switches to the other buffer
3. at the next packet boundary, the DMA ISR switches transmission to the new packet

As in the DxCore implementation, this avoids copying waveform data inside the ISR and prevents race conditions between the application code and the transmission engine.

Default packets are transmitted continuously:

- **idle packets** in normal mode
- **reset packets** in Service Mode

---

### AUX / RailCom cutout via a second PIO state machine

The AUX signal is generated by a second **PIO state machine**.

This state machine waits for an **IRQ event** from the DCC state machine. That IRQ is generated exactly when the DCC state machine encounters the packet-start marker. In this way, the start of the AUX RailCom cutout is synchronized directly to the packet boundary inside PIO hardware.

The AUX PIO program then generates the complete RailCom cutout pulse sequence autonomously:

1. wait for IRQ from the DCC state machine
2. apply a fixed delay
3. set the AUX pin active for the cutout interval
4. return the AUX pin to its normal level

Because both the DCC timing and the AUX cutout timing are generated inside the PIO subsystem, the synchronization does not depend on CPU interrupt latency.

This is the main architectural difference compared with implementations that start the AUX cutout from a timer ISR or DMA ISR.

---

### PIO configuration details

The implementation uses one selected PIO block:

- `pio0` in the current code

On that PIO block, two unused state machines are claimed:

- one for DCC
- one for AUX

The DCC state machine is configured to:

- output on the selected DCC pin
- run at **1 MHz**
- shift 32-bit symbols from the TX FIFO
- automatically pull new data from the FIFO

The AUX state machine is configured to:

- output on the selected AUX pin
- run at **1 MHz**
- respond to IRQ 0 from the DCC state machine

The clock divider is derived from `F_CPU` so that the effective PIO instruction rate becomes 1 µs per instruction.

---

### RailCom on the DCC signal

RailCom can also be generated directly in the DCC waveform itself.

In that case, a dedicated RailCom gap symbol is inserted at the start of the packet. As with the other drivers, the preamble is shortened accordingly so the total packet structure remains consistent.

If RailCom is enabled on AUX, packet-start markers are also inserted so that the AUX state machine remains synchronized with the DCC packet boundaries.

---

### Service Mode handling

Service Mode is handled in the same packet scheduler as normal packets.

In Service Mode:

- the preamble is longer
- reset packets replace idle packets as the default stream
- Service Mode packets can be retransmitted automatically

As in the other implementations, repetition is managed in the ISR-side scheduler and does not require bit-level CPU involvement.

---

### Pin inversion and AUX polarity

The RP2040 / RP2350 implementation uses `gpio_set_outover()` to invert output polarity.

This works not only for normal GPIO output, but also for signals driven by PIO. As a result:

- the DCC signal can be inverted without changing the waveform generation logic
- the AUX enable polarity can also be inverted independently

One point to note is that some GPIO / PIO initialization calls reset the inversion logic. For that reason, the code reapplies inversion after reattaching the AUX pin to PIO.

---

## Signal quality

The **DCC signal** is generated entirely by **PIO hardware**, with **DMA** feeding precomputed symbols into the PIO TX FIFO. Once transmission has started, the CPU is not involved in waveform timing. As a result, the DCC signal is fully deterministic and shows **no jitter**.

The **RailCom cutout on the AUX pin** is also generated entirely by **PIO hardware**. It is synchronized to the DCC packet boundary by means of a **packet-start marker** and a **PIO-to-PIO IRQ**, not by a CPU interrupt or timer callback. This means the AUX cutout is also exactly aligned and shows **no jitter**.

In practice, this gives the RP2040 / RP2350 implementation the most stable signal of all currently implemented drivers:

- DCC timing is exact
- AUX RailCom timing is exact
- synchronization between both signals is exact
- **jitter is 0**

This makes the RP2040 / RP2350 backend the most stable of the available driver implementations.
